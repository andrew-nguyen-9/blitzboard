# v2.2 — ideas & follow-ups for the backlog

Captured during the Scoring & Value-Engine phase. Not committed work — candidates for
later phases. (Carries forward the still-open items from `v2.1-ideas.md` / `v2.0-ideas.md`.)

## Tuning — the v2.4.3 backtest gate (already on the books)

- **All v2.2 knobs are placeholders.** `DISCOUNT_K` (f(ρ)=ρ^k, =1.0), `STREAMER_PCT`
  (=0.60), `MC_VOL_GAIN` (=0.60), and the predictability blend weights/`τ`/structural
  priors are configured constants chosen for sane behaviour and unit-tested for
  *direction* (monotonicity, ordering, calibration improvement) — **not** fit to a target.
  v2.4.3 tunes them on the 2021–2025 backtest (hold out 2025), reporting CIs not point
  wins. The unit tests are the regression guard while tuning.
- **`f(ρ)` form.** `ρ^k` is the simplest monotone discount; the backtest may prefer a
  gentler shoulder (e.g. `ρ^k` only below a ρ threshold, or a logistic) so stable offense
  takes ~no haircut while K/DEF still collapse. Keep it explainable.

## Predictability signal — refinements

- **TD/turnover share uses season stat lines, which are 0 for K/DST** (`score_stats`
  returns 0 for those positions), so K/DST ρ rests almost entirely on the structural
  prior. That's the intended (low) outcome, but once K/DST have real per-stat history
  (FG-distance buckets, D/ST events), feed those into `td_turnover_share` so a genuinely
  steady kicker can earn a *slightly* higher ρ from data, not just the prior.
- **Rank-stability prior is global, not era-aware.** `_rank_stability` pools all
  consecutive-season pairs in the store; a position whose volatility changed over time
  (e.g. TE) gets a blended prior. Consider weighting recent seasons more.
- **Single-season players get reliability=1.0** (no variance observable) before shrinkage.
  Shrinkage hides it, but a more honest prior-variance could replace the 1.0 default.

## Monte Carlo — distribution realism

- **Normal sampling is symmetric; real fantasy outcomes are right-skewed** (capped floor at
  ~0, long ceiling). The widening fixes *width* but not *shape*. A log-normal or skew-normal
  draw (or sampling the empirical residuals from the backtest) would calibrate the tails
  better — revisit when v2.4 has realized residuals to fit.
- **`MC_VOL_GAIN` is a single global gain.** Position-specific gains (K/DEF wider than QB)
  may calibrate better; the calibration harness can grade per-position once real outcomes
  exist.
- **Grade calibration on real data.** `calibration_check.py` runs on a synthetic world; wire
  `pit_values` to (emitted MC σ, realized season points from `player_stats_history`) for a
  real reliability diagram after a full v2 season — and surface the per-position error.

## Engine coherence / UX

- **Populated card needs a `monte_carlo` pipeline run.** The card morph + ridge are wired
  and tested, but the live view only fills once `value_engine_run.py --engine monte_carlo`
  has written rows and the predictability migration is applied. Schedule both with the v2.3
  deploy so the toggle isn't half-empty in production.
- **Ridge reconstructs σ from P90/P10** (`(boom−bust)/2.5631`) assuming normality; if MC
  moves to skewed draws, persist a few percentiles (P10/P50/P90) on `player_value` and shape
  the ridge from those instead of a re-derived σ.

## Infra / CI debt (surfaced this phase)

- **Pipeline tests aren't in CI.** `selftest.py` + `tests/test_*.py` run locally only; CI
  gates the frontend. Add a Python job (3.9, `pip install -r requirements.txt`, run the
  suites) so the modeling layer is gated like the frontend. Small, high-value.
- **Migrate-first has no automation.** The `predictability` column had to be hand-applied
  before deploy. A lightweight `db/migrations` runner (or Supabase migration tracking) would
  make additive columns part of the deploy, not a manual gate. Relevant from v2.3 on (every
  phase adds columns).
