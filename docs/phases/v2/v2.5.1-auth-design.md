# v2.5.1 — Supabase Auth + account model (design)

**Segment:** v2.5.1 · **Branch:** `v2.5-auth` (off `v2`) · **Date:** 2026-06-25
**Refines:** `docs/phases/v2/v2.5-auth-security.md` · **Spec:** `docs/security/SECURITY.md`, `docs/security/MULTI_LEAGUE.md`

> **Design decision that diverges from the phase doc:** the phase plan and `SECURITY.md`
> specify **Auth.js (NextAuth)**. After brainstorming the `auth.uid()`/RLS interplay, we are
> deliberately using **Supabase Auth (GoTrue)** instead. Rationale below; the divergent docs are
> amended as part of this segment.

---

## 1. Why Supabase Auth (not Auth.js)

The hard acceptance gate for this phase is **RLS isolation proven by test** — user A cannot
read/write user B's rows — keyed to `auth.uid()`. `auth.uid()` is a Postgres function that reads
`request.jwt.claims ->> 'sub'`; it is only populated when a request carries a JWT signed with the
project's Supabase JWT secret.

- **Auth.js path:** manages its own session cookie and does *not* mint a Supabase JWT. Making
  `auth.uid()` work requires minting a short-lived `SUPABASE_JWT_SECRET`-signed token per request
  (the `@auth/supabase-adapter` bridge), plus an edge-runtime crypto split (native hashing/`pg`
  can't run in Auth.js edge middleware), plus the credentials-provider-forces-JWT-sessions seam.
  Three avoidable seams in security-critical code.
- **Supabase Auth path:** `auth.uid()` is **native**. Google OAuth + email/password + reset +
  verification are built in. Password hashing is delegated to GoTrue. RLS is trivially correct.

We accept one trade-off: **password hashing is GoTrue's bcrypt**, not Argon2id. `SECURITY.md`
explicitly allows "bcrypt (cost ≥ 12) if Argon2 isn't available in the runtime" — GoTrue's bcrypt
is that sanctioned fallback. A bonus: the app never receives the plaintext password (it goes
browser → GoTrue over TLS), so the "no plaintext stored/logged" criterion holds **by construction**.

## 2. Architecture

Add **`@supabase/ssr`** (the App-Router cookie bridge) alongside the existing
`@supabase/supabase-js`. Three client factories with clear boundaries:

| Factory | Runs in | Key | Purpose |
|---------|---------|-----|---------|
| `getSupabase()` *(existing, unchanged)* | browser / public reads | anon | public player universe; stays null-safe (`null` offline) |
| `getServerSupabase()` *(new)* | Server Components, Route Handlers, Server Actions | anon + **user JWT from httpOnly cookie** | authenticated reads/writes; `auth.uid()` populated → **RLS enforced at the DB** |
| service-role *(added in v2.5.3, not here)* | pipeline / server only | service-role | bypasses RLS; **never** in the client bundle |

- **`frontend/middleware.ts`** refreshes the auth session cookie on each request (canonical
  `@supabase/ssr` pattern). Session is stored in **httpOnly, Secure, `SameSite=Lax`** cookies set
  server-side. `SameSite=Lax` (not `Strict`) is required so the OAuth redirect-back carries the
  cookie. This satisfies the v2.5.1.1 gate: the cookie is not script-readable.
- **Auth UI:** `/login` and `/signup` routes — a Google button (`signInWithOAuth`) and an
  email/password form (`signInWithPassword` / `signUp`). `app/auth/callback/route.ts` exchanges
  the OAuth `code` for a session. A **redirect allowlist** prevents open-redirects on `next=`.
- **Offline-first preserved:** when `NEXT_PUBLIC_SUPABASE_*` are absent, the auth UI renders a
  graceful "sign-in unavailable offline" state, mirroring the existing null-safe degradation. The
  app still builds and renders with no keys.

### Client/server boundary rules
- Anon key + URLs are the only Supabase values shipped to the browser (`NEXT_PUBLIC_*`).
- `getServerSupabase()` is import-guarded to server contexts; it reads/writes cookies via the
  `cookies()` API and never runs in a Client Component.
- No service-role key anywhere in v2.5.1.

## 3. Data model (v2.5.1.3)

```
public.accounts
  user_id      uuid PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE
  display_name text
  email        text
  prefs        jsonb NOT NULL DEFAULT '{}'::jsonb   -- a11y/theme sync from v2.0
  created_at   timestamptz NOT NULL DEFAULT now()
```

- **Row creation:** a `handle_new_user()` `SECURITY DEFINER` trigger on `auth.users` inserts the
  matching `accounts` row on signup. The client never inserts its own account row (never trust a
  client-supplied `user_id`).
- **RLS:** enabled; `select`/`update`/`delete` policies `using (user_id = auth.uid())`. No client
  `insert` policy (the trigger owns inserts).
- **prefs sync:** read on load via `getServerSupabase()`; written on change via a server action
  that re-checks the session. `prefs` shape validated with **zod** before write.

## 4. Migrations & hand-apply gates

The user has **no Supabase CLI/DDL access**, so every change is hand-applied before merge.

### Gate A — SQL migration (idempotent, timestamp-prefixed)
`db/migrations/<ts>_v2.5.1_accounts.sql`:
- `create table if not exists public.accounts (...)`
- `alter table public.accounts enable row level security;`
- `drop policy if exists ...; create policy ...` for select/update/delete on `auth.uid()`
- `create or replace function public.handle_new_user() ... security definer ...`
- `drop trigger if exists on_auth_user_created on auth.users; create trigger ...`

Matches the house style in `db/migrations/*` (intent-comment header, `if not exists` /
`drop ... if exists` for re-runnability).

### Gate B — Supabase dashboard checklist (NOT SQL — GoTrue config)
A click-path the user must apply by hand, delivered with the segment:
1. **Authentication → Providers → Email:** enable; set "Confirm email" on.
2. **Authentication → Providers → Google:** enable; paste Google OAuth client id + secret.
3. **Google Cloud console:** authorized redirect URI → `https://<project>.supabase.co/auth/v1/callback`.
4. **Authentication → URL Configuration:** Site URL + additional redirect URLs (local + Vercel).
5. *(v2.5.1.2, prod only)* **custom SMTP → Resend** once a domain is verified.

**No migration or provider is assumed live until the user confirms it is applied.**

## 6. Testing (TDD, honest about boundaries)

`vitest` covers what is pure/deterministic without a live project:
- `prefs` **zod** schema (valid/invalid shapes, defaults).
- `getServerSupabase()` **cookie options** — assert `httpOnly`, `secure`, `sameSite: 'lax'`.
- **redirect-allowlist** validation (rejects external/open redirects on `next=`).
- **offline null-safe** degradation (no env → auth UI shows unavailable state, no throw).

Not unit-testable here (covered by Playwright + manual QA): live Google OAuth round-trip,
GoTrue email delivery. The **RLS cross-user A-vs-B test is v2.5.2's hard gate**; v2.5.1 scaffolds
the integration harness (two-user fixture, two JWTs) that v2.5.2 will assert against.

Per-task DoD (`DEFINITION_OF_DONE.md`): `npm run build` + `tsc --noEmit` clean; new logic tested;
**no service-role key reachable from the browser**; commit `v2.5.1.<t> <scope>: <summary>` with no

`accounts`. The full cross-user RLS proof lands in v2.5.2 but `accounts` must already be RLS-correct.
