# Phase v2.5 — Auth, Accounts & Security

**Branch:** sub-branches `v2.5-*` off `v2` · **Covers:** #5 (login + saved creds), #7
(security/hashing/isolation), #9 (multi-league + rules import)
**Depends on:** v2.0 · **Unblocks:** gated tabs (v2.6)
**Spec:** `docs/security/SECURITY.md`, `docs/security/MULTI_LEAGUE.md`

## Goal

Stand up real, secure accounts (Google + email/password via Supabase Auth on Supabase Postgres),
an encrypted ESPN/Sleeper credential vault, RLS isolation proven by test, and the multi-league
data model with rules import.

## Segments

### v2.5.1 — Supabase Auth + account model (`v2.5-auth`)
> **Revised from Auth.js → Supabase Auth (GoTrue).** Native `auth.uid()` is the simplest, strongest
> path to the RLS gate. Full rationale + design: `docs/phases/v2/v2.5.1-auth-design.md`.
- **v2.5.1.1** Supabase Auth (GoTrue) with Google OAuth + email/password via `@supabase/ssr`;
  httpOnly/Secure/SameSite=Lax session cookies (server-managed), HTTPS-only. *Accept:* sign in via
  Google and via email works; session cookie is httpOnly (not script-readable).
- **v2.5.1.2** Password hashing by GoTrue (bcrypt — Argon2id not available in GoTrue; bcrypt is the
  SECURITY.md fallback; app never sees plaintext); password reset + email verification. *Accept:*
  no plaintext stored/logged; reset flow works.
- **v2.5.1.3** `accounts` table + prefs (a11y/theme sync from v2.0). *Accept:* account row
  created on signup; prefs persist server-side.
- *Files:* `frontend/lib/supabase/server.ts`, `frontend/middleware.ts`, `frontend/app/login`,
  `frontend/app/signup`, `frontend/app/auth/**`, `frontend/app/actions/{auth,prefs}.ts`,
  `frontend/lib/auth/*`, `db/migrations/*`.

### v2.5.2 — RLS isolation (`v2.5-rls`)
- **v2.5.2.1** RLS on every user table (`accounts`, `user_leagues`, `credential_vault`) keyed
  to `auth.uid()`; server-side authz on mutations. *Accept:* automated test proves user A
  cannot read/write user B's rows.
- **v2.5.2.2** Confirm service-role key absent from client bundle; anon key only does
  public-read. *Accept:* bundle scan clean; RLS denies cross-user access with anon key.
- *Files:* `db/migrations/*` (policies), `frontend/lib/queries.auth.ts` (new), tests.

### v2.5.3 — Encrypted credential vault (`v2.5-vault`)
- **v2.5.3.1** `credential_vault` with AES-256-GCM envelope encryption (master key server/KMS
  only); write-once via a server action; return only a masked "connected ✓". *Accept:*
  plaintext never returns to the client; ciphertext at rest; RLS-isolated.
- **v2.5.3.2** Pipeline decrypts transiently server-side to call ESPN/Sleeper; expiry
  re-prompt; disconnect = hard delete. *Accept:* pipeline can use a vaulted credential;
  disconnect removes it.
- *Files:* `frontend/app/actions/credentials.ts` (new), `pipeline/league_sync.py`, `db/migrations/*`.

### v2.5.4 — Multi-league + rules import (`v2.5-leagues`)
- **v2.5.4.1** `user_leagues` + multi-tenant `league_rules`; league switcher; default league.
  *Accept:* a user holds ≥2 leagues, switches between them, RLS-isolated.
- **v2.5.4.2** Import: Sleeper (public API) + ESPN (vaulted creds) → parse settings into
  `league_rules` JSONB; **confirm/edit** screen before it drives value. *Accept:* superflex,
  distance-K, yardage-D/ST, FAAB detected; user confirms before activation.
- **v2.5.4.3** Manual rules editor fallback. *Accept:* a league with no auto-pull can be set up manually.
- *Files:* `frontend/app/leagues/**` (new), `frontend/lib/leagueImport.ts`, `pipeline/league_sync.py`.

### v2.5.5 — Header/transport hardening baseline (`v2.5-headers`)
- **v2.5.5.1** HSTS, CSP, secure headers, CSRF protection, zod validation on server boundaries,
  rate-limit auth/write endpoints. *Accept:* securityheaders-style scan clean; CSRF blocked.
- *Files:* `next.config.mjs`/middleware, `frontend/lib/validation.ts` (new).

## Test / QA plan

- Auth flows: Google + email, reset, verify, logout invalidation, session fixation, OAuth
  state/nonce, CSRF.
- RLS cross-user test (A vs B) automated; bundle secret scan; header/TLS scan.
- Vault round-trip: no plaintext to client/logs; pipeline decrypt works; disconnect deletes.
- Multi-league: connect/import/confirm/switch, RLS-isolated; rules detection correctness.

## Risks & mitigations

- *Auth ↔ RLS integration seams* → Supabase Auth makes `auth.uid()` native, so RLS is the source of
  truth for data isolation (don't rely on app checks alone); no JWT-minting bridge needed.
- *ESPN cookie rotation* → expiry detection + re-prompt; graceful "reconnect" UX.
- *Key management* → master key in server/KMS env only; never in repo or client; documented rotation.
